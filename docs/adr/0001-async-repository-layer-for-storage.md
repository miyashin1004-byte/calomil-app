# データ層をPromiseベースのリポジトリ層で抽象化し、localStorageを直接扱わない

現状は各画面のコードが `localStorage.getItem/setItem` を直接呼んでおり、保存先が画面側から丸見えになっている。今回はバックエンド化自体はスコープ外だが、機種変更・ブラウザデータ消去で記録が消えるリスクを踏まえ、将来サーバーAPIに差し替えやすくしておきたい。

そのため、画面側は「今日の食事を取得する」「食事記録を保存する」といった意味だけを持つリポジトリ関数を呼び、その内部実装(localStorageかfetchか)を意識しない形にする。これらの関数は最初からPromiseを返す非同期関数として実装する。今はlocalStorage実装なので同期的に一瞬で解決するが、こうしておくことで将来API通信に差し替える際、呼び出し側(画面コード)を書き換えずに済む。同期のまま作ると、後で非同期化する際に呼び出し側全体を書き直す必要が生じるため、最初から非同期の形に揃えておく方が移行コストが低いと判断した。
